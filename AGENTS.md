# Agent 協作準則（LLM）

以下準則為 Agent 在協力開發專案時必須遵守的基本行為，詳細程式碼開發規則請參照：`_docs/code-style/SKILL.md`

## 1. 動手前先想清楚

**不要臆測。不要藏著困惑。把取捨攤開來說。**

實作前：

- 明確說明你的假設；不確定就問。
- 若有多種解讀，列出來——不要默默選一種。
- 若有更簡單的做法，要說出來；必要時提出異議。
- 若有不清楚之處，先停下來，點出困惑之處，再發問。

## 2. 簡潔優先

**用最少程式碼解決問題。不做投機性擴充。**

- 不做超出需求的功能。
- 不為只用一次的程式碼抽象。
- 不加入未被要求的「彈性」或「可設定性」。
- 不為不可能發生的情境寫錯誤處理。
- 若寫了 200 行其實 50 行就夠，就重寫。

自問：「資深工程師會不會覺得過度複雜？」若是，就簡化。

## 3. 手術式變更

**只動該動的。只清理自己造成的殘留。**

編輯既有程式碼時：

- 不要「順便」改相鄰程式碼、註解或格式。
- 不要重構沒壞的東西。
- 符合既有風格，即使你會用不同寫法。
- 若發現無關的死碼，提及即可——不要擅自刪除。

若你的變更產生孤兒程式碼：

- 刪除**你的變更**導致未使用的 import／變數／函式。
- 除非被要求，不要刪除既有的死碼。

檢驗標準：每一行變更都應能直接對應到使用者的請求。

## 4. 以目標驅動執行

**定義成功標準。反覆執行直到可驗證。**

把任務轉成可驗證的目標：

- 「加驗證」→「先為無效輸入寫測試，再讓測試通過」
- 「修 bug」→「先寫能重現 bug 的測試，再讓測試通過」
- 「重構 X」→「重構前後測試皆須通過」

多步驟任務請簡述計畫：

```
1. [步驟] → 驗證：[檢查方式]
2. [步驟] → 驗證：[檢查方式]
3. [步驟] → 驗證：[檢查方式]
```

明確的成功標準能讓你獨立迭代；模糊的標準（例如「弄好就好」）會需要不斷追問。

## 5. 紀錄執行問題

當任務過程受到 sandbox 限制而無法繼續執行時，應該使用這個 skill：
`_docs/task-problem-log/SKILL.md`

---

**本準則有效的跡象：** diff 中不必要的變更少、因過度複雜而重寫的次數少，且釐清問題發生在實作之前而非犯錯之後。
